# 策略优化 Skill 体系 — 讨论记录

## 一、整体定位

面向美团类本地生活/电商企业的内部运营场景（销售、补贴、投放、用户增长等），通过分析 Agent 调用一组 skill，实现策略的优化，带来实际业绩提升。

核心能力方向：策略精细化、新策略挖掘。

与已有「业务指标异动分析」skill 的关系：异动分析是"向后看"（诊断过去），这组新 skill 是"向前看"（优化未来）。

## 二、设计理念

**一套框架结构 + 可插拔的业务细节**

- 先在投放场景验证
- 可迁移到其他场景（补贴、用户运营、销售等），只需调整 skill 内部业务细节，框架结构不变

## 三、Skill 清单（5个）

| # | Skill 名称 | 职责 |
|---|-----------|------|
| 1 | 业务背景对齐 | 明确业务目标、指标体系、围栏约束、能力约束、重点动作导向 |
| 2 | 策略库 | 基于业务逻辑建立策略地图；通过历史数据冷启动；持续迭代更新 |
| 3 | 策略优化 | 基于业务目标和约束，结合策略库信息，制定最优策略方案 |
| 4 | 策略挖掘 | 通过大量数据/信息，探索挖掘更有效的新策略方案 |
| 5 | 效果评估 | AB testing 结果分析、DID 评估等 |

### Skill 间协作关系

```
业务背景对齐 → 策略优化（基于已有策略）→ 执行
     ↓                                    ↓
   策略库 ←──── 策略挖掘 ←──── 效果评估
```

效果评估结论反哺策略库，形成迭代闭环。

## 四、铁律

> **所有 wiki 内容的创建和修改，必须经过用户确认。Agent 绝不能自行编造或擅自修改。**

## 五、Skill 1：业务背景对齐 — 详细设计

### 5.1 目的

把一个模糊的"提高业绩"锁定成一个**有明确目标、有清晰边界、有可行动作空间**的问题定义。

### 5.2 核心输出要素

#### 围栏约束（多目标权衡的结构化表达）

实际业务不是单目标，常见目标维度：规模、质量、效率、体验、风险。

处理方式：
- 选一个维度做"目标"（如规模最大化）
- 其他维度变成"围栏约束"（效率≥X、质量≥Y）
- 最终产出 = 项目的**指标体系**（目标指标 + 围栏指标 + 各自阈值）

投放场景示例：
- 目标：投放带来的新客/GMV 规模最大化
- 围栏-效率：预算约束 → ROI 不能低于某阈值
- 围栏-质量：不能是薅羊毛用户 → 单用户 GMV ≥ X

#### 能力约束（策略的"动作空间"）

可以在什么范围内做策略：
- 差异化颗粒度：能否分人群/分时/分地域，精细到什么程度
- 可选动作集：投放素材、补贴形式、触达通道有哪些选项
- 执行限制：人力/系统决定的策略调整频率和响应速度

#### 重点动作导向

上层战略意图对这个项目的方向约束：
- 比如"今年重点做下沉市场"或"优先提升复购而非拉新"
- 在更大层面目标下，希望这个项目/业务在什么方向上带来更多增长或业绩改善

### 5.3 输入方式

灵活的信息收集：
- 通过对话逐步引导用户明确
- 用户也可直接提供文档（markdown）
- 信息不足时主动追问补全

### 5.4 输出方式

参考 LLM Wiki 模式（Karpathy）：

**完整文档**（给人看）+ **多个子文档**（给 skill 按需调用）

```
project-wiki/
├── index.md                ← 索引，全局入口
├── overview.md             ← 完整业务背景（人读）
├── metrics.md              ← 指标体系（目标+围栏）
├── constraints.md          ← 能力约束（动作空间）
├── strategic-direction.md  ← 重点动作导向
└── log.md                  ← 变更记录
```

- 完整文档：项目全景 wiki，人可以通读理解全局
- 子文档：拆开的独立页面，下游 skill 按需读取相关页面，不需要每次加载全部上下文

### 5.5 维护模式

两个场景：
1. **首次创建**：通过对话引导，逐步建立项目 wiki，每一步都跟用户确认
2. **持续迭代**：随业务推进（如每周策略更新），根据用户对话和反馈提出修改建议，用户确认后才写入

## 六、跨 Skill 协作机制

### 6.1 信息传递方式：共享文件（方案A）

所有 skill 通过读写同一套文件进行协作，不依赖 Agent 上下文传递。

### 6.2 项目文件结构

每个项目一套独立目录，不同项目完全隔离：

```
{项目名}/
├── wiki/                    ← 业务背景对齐 skill 输出
│   ├── index.md
│   ├── overview.md
│   ├── metrics.md
│   ├── constraints.md
│   ├── strategic-direction.md
│   └── log.md
├── strategy-library/        ← 策略库 skill 管理
│   ├── framework.md         ← 策略框架（要素维度+值空间）
│   ├── mutual-exclusion.md  ← 互斥关系
│   ├── strategies.csv       ← 策略效果数据（全量）
│   └── log.md
└── outputs/                 ← 策略优化/挖掘的输出
    ├── ...
```

### 6.3 数据格式

- wiki 文档：Markdown（人可读 + 模型可读）
- 策略效果数据：CSV 存储（人用 Excel 打开），模型使用时用 Python 按需筛选后转为 markdown/文本片段消费
- 兼顾人友好和模型友好

### 6.4 各 Skill 的读写关系

| Skill | 读取 | 写入 |
|-------|------|------|
| 业务背景对齐 | — | wiki/ |
| 策略库 | wiki/ | strategy-library/ |
| 策略优化 | wiki/ + strategy-library/ | outputs/ |
| 策略挖掘 | wiki/ + strategy-library/ + 原始数据 | outputs/ → 确认后写入 strategy-library/ |
| 效果评估 | wiki/metrics.md + 明细数据 | strategy-library/strategies.csv |

### 6.5 调用方式

skill 支持多种调用模式，不限定：
- 被 Agent 自动编排调用
- 被用户直接触发
- 混合模式（Agent 编排 + 用户确认）

### 6.6 Skill 设计要求（支持多种调用模式）

1. **自包含**：不依赖"上一步 Agent 传了什么上下文"，自己能从共享文件中读取所需信息
2. **入口清晰**：明确需要什么输入、从哪里读、输出写到哪里
3. **可独立运行**：单独调用任何一个 skill 都能正常工作（前提是依赖的文件存在）
4. **缺依赖时提示**：如果依赖的文件不存在（如直接调策略优化但没做过业务背景对齐），skill 应提示用户需要先完成前置步骤

## 七、通用框架设计

### 7.1 核心原则

**框架层固定，业务细节层可插拔。换场景时只需重新跑业务背景对齐生成新配置，skill 的 prompt/逻辑不需要调整。**

### 7.2 分层划分

**不变的（框架层）**：
- 5 个 skill 的职责分工和协作流程
- 文件结构（wiki/ + strategy-library/ + outputs/）
- 策略库的两层结构（框架+效果）
- 策略优化的算法逻辑（整数规划求解）
- 效果评估的方法论（AB/同比/DID）
- 各 skill 的工作流程步骤

**要改的（业务细节层，通过配置文件管理）**：
- 策略框架的要素维度和值空间
- 指标体系的具体指标
- 互斥规则
- 策略挖掘的具体模板和假设方向
- 外部环境的具体指标

### 7.3 配置文件管理方式

- 配置文件由 Agent 和用户对话后**自动生成**，用户只负责确认和管理，不需要手动编写
- 流程：业务背景对齐 skill 对话引导 → 自动生成配置文件 → 用户确认 → 配置生效
- 配置文件既是 wiki 的一部分（人可读可管理），也是下游 skill 的运行参数

### 7.4 换场景流程

例如从投放场景扩展到补贴场景：
1. 重新跑一次"业务背景对齐 skill"，对齐补贴业务的目标、指标、策略要素
2. Agent 自动生成新项目的配置文件
3. 用户确认
4. 其他 4 个 skill 无需修改，读取新配置即可运行

### 7.5 Skill 编写要求

**skill 内部不能硬编码任何业务细节**（如"人群""渠道""ROI"这些词），而是引用配置文件中的变量。

示例：
- ❌ "读取人群×渠道的策略效果"
- ✅ "读取策略框架中定义的要素维度对应的策略效果"
