# 策略优化 Skill — 讨论记录

## 一、核心定位

基于业务目标和约束，结合策略库中信息（含实测和拟合效果），制定最优策略组合方案。

本质是一个**约束优化问题**。

## 二、输入

- 业务目标（如增量用户数最大化）
- 约束条件（总预算、ROI≥X、质量围栏等）
- 当前外部环境（一般为供需、竞争，但不是必须输入，如果没有则认为外部环境不变）
- 策略库（历史策略效果 + 拟合策略效果 + 各自置信度）

## 三、核心逻辑

### 3.1 问题本质

**带约束的组合优化 / 整数规划问题**：从 N 个离散策略中，选出最优子集，在约束下使目标最大化。

关键认知：
- 策略是离散的有限组合（人群×渠道×时段×补贴档位×... = 一个具体策略）
- 资源/投入规模（如补贴深度）本身是策略的一个要素维度，不是策略之外的独立变量
- 例如「人群A × 渠道B × 补贴5元」和「人群A × 渠道B × 补贴10元」是两个不同策略
- 维度多时策略组合数量爆炸（5维各5值 = 3125个策略），必须通过算法求解

### 3.2 算法步骤

1. 从策略库获取所有策略及其效果值（含实测和拟合，各有置信度）
2. 效果值选择逻辑：优先选外部环境匹配的记录，或最近一次效果数据
3. 将业务目标和围栏约束转化为数学表达
4. 加入策略互斥约束（从策略库获取）
5. 调用求解器（如 scipy.optimize / 整数规划求解器）求解
6. 将数学结果翻译回业务语言，按置信度分级输出

### 3.3 算法输入

| 输入 | 来源 | 说明 |
|------|------|------|
| N 个离散策略 + 各自效果值 | 策略库 | 含实测和拟合，缺失效果已在策略库中完成拟合填充 |
| 策略置信度 | 策略库 | 实测=高置信，拟合=中/低置信 |
| 策略互斥关系 | 策略库 | 哪些策略不能同时选（如同一用户不能命中两个补贴） |
| 目标函数 | 业务背景对齐 | 如增量用户数最大化 |
| 约束条件 | 业务背景对齐 | 总预算、ROI≥X、质量围栏等 |
| 当前外部环境 | 用户提供（非必须） | 用于筛选策略库中匹配的效果记录，缺失则认为环境不变 |

### 3.4 算法求解器选择

**问题形式化**：

```
决策变量：x_k ∈ {0, 1}，每个策略选或不选
目标：max Σ(x_k × effect_k)  （如总增量用户数）
约束：
  - Σ(x_k × cost_k) ≤ 总预算
  - Σ(x_k × revenue_k) - ROI阈值 × Σ(x_k × cost_k) ≥ 0  （比率约束转线性）
  - x_a + x_b ≤ 1（互斥策略不能同时选）
```

本质是 **0-1 整数线性规划**。

**场景特点**：
- 策略数量：维度交叉后几百~几千个
- 约束类型：预算上限、比率围栏（可转线性）、互斥约束
- 运行环境：Python
- 输出要求：可解释，业务人员能理解

**求解器方案对比**：

| 求解器 | 类型 | 适用规模 | 优点 | 缺点 |
|--------|------|---------|------|------|
| `PuLP`（推荐） | 建模库 + CBC求解器 | 千级策略 | 开源免费、建模语法可读、零额外依赖、互斥约束直接写、适合模板化 | 超大规模性能不如商业求解器 |
| `scipy.optimize.milp` | 混合整数线性规划 | 百~千级 | Python 内置 | 建模要写矩阵，代码可读性差 |
| `OR-Tools`（Google） | 组合优化 | 万级策略 | 功能强 | 规模用不到，引入不必要复杂度 |
| `Gurobi` / `CPLEX` | 商业级 | 百万级 | 性能最强 | 需付费 |

**初步推荐 PuLP**，理由：
1. 几千个 0-1 变量秒级出结果，规模完全够用
2. 代码本身像业务描述，LLM 生成的代码用户也能看懂
3. `pip install pulp` 即可，求解器内置
4. 换业务场景时只需改变量定义和约束条件，框架代码不变

**示例代码框架**：

```python
from pulp import *

prob = LpProblem("策略优化", LpMaximize)

# 决策变量：每个策略选或不选
x = {k: LpVariable(f"策略_{k}", cat='Binary') for k in strategies}

# 目标：增量用户数最大化
prob += lpSum(x[k] * effect[k] for k in strategies)

# 约束1：总预算
prob += lpSum(x[k] * cost[k] for k in strategies) <= budget

# 约束2：ROI围栏（比率约束已转线性）
prob += lpSum(x[k] * revenue[k] for k in strategies) - ROI_threshold * lpSum(x[k] * cost[k] for k in strategies) >= 0

# 约束3：互斥
for a, b in mutual_exclusive_pairs:
    prob += x[a] + x[b] <= 1

prob.solve()
```

### 3.5 关键设计决策

- **策略效果拟合在策略库 skill 中已完成**：策略优化 skill 拿到的每个策略都已有效果值，不需要再做拟合
- **边际递减已体现在离散策略中**：不同投入规模是不同策略，各有各的效果值
- **互斥关系需要在策略库中预先和用户明确**：作为算法的硬约束输入

## 四、输出：多版本方案

按置信度分级给出不同版本方案：

| 方案版本 | 策略构成 | 特点 |
|---------|---------|------|
| 高置信方案 | 主要由历史实测策略组成 | 效果确定性高，风险低 |
| 中置信方案 | 包含部分拟合策略 | 预期收益更高，有一定不确定性 |
| 高风险方案 | 包含较多未验证策略 | 潜在收益最大，风险也最大 |

用户可以根据风险偏好选择单一版本，或做组合（如 70% 预算走高置信，30% 试验高风险策略）。

## 五、输出颗粒度

每个方案版本都需要是具体可执行的：

- 具体策略明细（人群×渠道×时段×供给×补贴的具体值和预算分配）
- 与当前策略的对比：调整了什么
- 调整原因：为什么这样调
- 预期效果：量化的效果预估
- 置信度标注：每条策略标注是实测还是拟合，置信程度

示例输出：
```
【高置信方案】预期增量用户 +500，CAC 18元
  - 人群A（25-30岁女性）投放预算 10万 → 维持不变（实测 CAC 15元）
  - 人群B（30-35岁男性）投放预算 5万 → 增加至 8万（实测 CAC 16元，有余量）
  - 渠道：信息流维持，搜索增加 3万

【中置信方案】预期增量用户 +700，CAC 17元
  - 在高置信方案基础上：
  - 新增人群C（25-30岁男性）投放 5万（拟合 CAC 14元，基于相邻人群 A/B 数据，中置信度）
  - 调整原因：人群C与人群A年龄重叠、与人群B性别相同，拟合效果优于当前均值

【高风险方案】预期增量用户 +1000，CAC 15元
  - 在中置信方案基础上：
  - 新增晚间时段投放（从未测试，基于其他渠道晚间数据拟合，低置信度）
  - 新增素材类型D（从未使用，基于相似素材效果推断）
```

## 六、与其他 Skill 的关系

- **策略库**：提供策略效果数据（实测+拟合+置信度），策略优化 skill 读取使用
- **效果评估**：高风险策略上线后通过 AB test 验证，结果回填策略库，下次置信度提升
- **业务背景对齐**：提供目标、约束、能力边界等输入条件

## 七、关键原则

- 不是输出单一"最优解"，而是按置信度分级的多版本方案
- 让用户基于风险偏好做选择
- 所有方案必须是可直接执行的颗粒度
- 拟合策略必须标注置信度和拟合依据
