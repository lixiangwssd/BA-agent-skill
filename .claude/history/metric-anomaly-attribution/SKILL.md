# 指标异动归因分析

## 概览

使用此 skill 分析公司一个或多个指标的异常变化。最终输出应是一份有数据支撑的归因结论：指标是否真的发生异动，哪些维度对变化贡献最大，以及哪些业务解释被数据支持、被数据否定或仍需补充验证。

## 必要输入

分析前，先收集或推断以下信息：

- 指标名称、指标公式、统计粒度、负责人和业务含义。
- 当前周期和对比周期，例如日环比、周环比、同比、上线前后、实验组和对照组。
- 数据来源，例如数据表、查询结果、电子表格、看板导出或用户直接提供的数据。
- 可用于拆解的候选维度，例如地区、渠道、产品、用户 cohort、用户分群、商家、SKU、版本、平台、活动和生命周期阶段。
- 已知业务背景，例如功能上线、事故、流量变化、策略调整、季节性、节假日、营销活动、数据链路变化或埋点变化。

如果关键输入缺失，先基于明确假设继续分析，并把置信度标为低。只有当指标公式、对比周期或数据来源完全无法推断时，才向用户追问。

## 工作流程

### 1. 定义并验证异动

在解释异动前，先判断观察到的变化是否应被视为真实异动。

- 复述指标公式和对比周期。
- 比较绝对变化、相对变化，以及变化对业务影响的贡献。
- 如果有历史数据，检查近期基线波动。
- 检查季节性、日历效应、分母变化、样本量和数据新鲜度。
- 检查指标链路：埋点、ETL、看板筛选条件、去重逻辑、延迟到达数据和口径变更。
- 将结论归类为 `真实异动`、`正常波动`、`数据质量问题` 或 `暂无法判断`。

### 2. 拆解指标变化

将总指标变化拆解为不同维度的驱动因素。

- 对加总型指标，计算各分段的变化量和贡献占比。
- 对比率型指标，尽可能拆分分子效应、分母效应和结构变化效应。
- 对漏斗型指标，按阶段转化率、流量结构和流量规模拆解。
- 对多维分析，先从高层维度入手，再递归下钻到贡献最大的分段。
- 按对总变化的贡献度排序，而不是只看分段自身增长率。
- 同时保留方向和规模：某个分段可能增长很快，但如果基数很小，对总指标贡献可能有限。

### 3. 归因可能业务原因

把数据驱动因素转化为业务假设，并区分证据和猜测。

- 将贡献最高的维度映射到可能的业务机制。
- 查找可相互印证的指标：上游流量、下游转化、留存、客单价、取消、退款、激活、供给、库存、延迟、错误率和曝光。
- 区分内部原因和外部原因，例如产品上线、定价、运营、营销、供给、宏观环境、节假日、竞品变化或数据采集变化。
- 对每个假设说明支持证据、反向证据、缺失证据和置信度。
- 当数据只支持相关性时，不要直接宣称因果关系。

### 4. 输出最终分析

使用以下结构：

1. `结论`：用一段话说明异动状态、主驱动因素和最可能原因。
2. `异动定义`：说明指标公式、对比周期、阈值或基线，以及验证结果。
3. `变化概览`：说明总变化、绝对变化、相对变化和业务影响。
4. `贡献拆解`：用排序表展示维度或分段的基准值、当前值、变化量、贡献度和备注。
5. `归因假设`：列出可能原因、支持证据、反向证据、待补充验证和置信度。
6. `建议下一步验证`：给出具体查询、看板、实验或需要沟通的业务方。

## 分析规则

- 始终区分 `指标变化`、`维度贡献` 和 `业务原因`。
- 排序驱动因素时，优先看对总变化的贡献度，而不是原始分段增长率。
- 不要过度拟合单一维度；检查相邻维度中是否也出现同方向变化。
- 当数据缺失、样本量较小或指标定义模糊时，明确说明不确定性。
- 如果用户提供多个指标，先分别分析每个指标，再统一解释共同驱动因素和冲突之处。
- 如果同一维度解释了多个指标的变化，将其作为跨指标驱动因素重点呈现。
